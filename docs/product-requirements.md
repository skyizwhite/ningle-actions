# プロダクト要求定義書 (Product Requirements)

## 1. プロダクトビジョンと目的

### ビジョン
Common Lisp の軽量 Web フレームワーク **ningle** において、サーバーサイドの処理を「アクション」として宣言的に定義し、クライアント（特に htmx）から型安全かつ最小の記述で呼び出せるようにする拡張ライブラリ **ningle-actions** を提供する。

Next.js の Server Actions が「サーバー上で動く関数を、エンドポイントの手書き配線なしにクライアントから呼べる」体験をもたらすのと同様に、ningle-actions は ningle 上で次の体験を実現する。

> サーバー関数を1つ定義すれば、エンドポイントの生成・登録・ディスパッチ・URL 参照が自動で配線され、URL を一切意識せずに済む。

### 目的
- ningle における「htmx リクエストを受け取り、処理し、HTML を返す」までの URL 設計・ルート登録の定型コード（ボイラープレート）を排除する。
- アクションを「名前で呼べる関数」として扱えるようにし、エンドポイント URL の設計・記憶・手書き配線をなくす。
- ハイパーメディア駆動（htmx）アーキテクチャと自然に噛み合う、HTML フラグメントを返すアクションを第一級にする。
- リクエスト値の扱いは ningle の `params` 流儀を踏襲し、独自の引数規約を持ち込まない（型強制・検証は将来の任意機能）。
- ビュー層（HTML 生成）には依存せず、あらゆる HTML 生成手段（文字列、Spinneret、cl-who 等）と組み合わせ可能にする。

### 非目的（このプロダクトがやらないこと）
- 独自のクライアント JS ランタイムやバンドラの提供（htmx 等の既存ハイパーメディアライブラリに委ねる）。
- HTML テンプレートエンジンそのものの提供。
- ningle / Clack / Lack の置き換え。あくまで ningle の上に乗る薄い拡張。
- フルスタック SPA フレームワーク化。

## 2. ターゲットユーザーと課題・ニーズ

### ターゲットユーザー
1. **ningle で Web アプリを書く Common Lisp 開発者**
   - サーバーサイドレンダリング + htmx でアプリを構築している、または構築したい。
2. **ハイパーメディア駆動アーキテクチャの実践者**
   - 重い SPA を避け、サーバー主導で UI を更新したい。

### 課題・ニーズ
| # | 課題 | ニーズ |
|---|------|--------|
| C1 | ルートごとに `(setf (ningle:route app ...))` を手書きし、URL を設計・管理する手間がある | アクション定義から自動でエンドポイントを生成し、URL を意識せず参照したい |
| C2 | htmx 連携時、HTML フラグメントや `HX-*` レスポンスヘッダの取り回しが煩雑 | フラグメント返却・htmx レスポンス制御を簡潔に書きたい |
| C3 | アクション群が複数モジュールに散在し、URL 名前空間の管理が煩雑 | アクションを一括で本体へマウントし、URL 名前空間を分離したい |
| C4 | 文字列パラメータの型変換・入力バリデーションを各ハンドラで手書きしている（将来課題） | 宣言的な型強制・検証を将来的に提供したい（MVP では利用者責務） |

## 3. 主要な機能一覧

| ID | 機能 | 概要 | MVP |
|----|------|------|-----|
| F1 | アクション定義マクロ（defaction） | `(defaction NAME METHOD (PARAMS) &body)` でサーバー関数を定義する。本体は ningle 同様に `params` を受け取る | ✅ |
| F2 | 自動登録・ディスパッチ | アクションをレジストリへ登録し、単一ルート `/:action_id` で `action_id` からハンドラへディスパッチする | ✅ |
| F3 | エンドポイント関数生成 | `defaction` がアクション登録と同時に、エンドポイント URL を返す関数を定義する（利用者は URL を意識しない）。キーワード引数を渡すとクエリパラメータとして URL に付加される | ✅ |
| F4 | シングルトンアクションアプリ | ライブラリがシングルトンのアクションアプリ `*actions-app*` を提供し、`defaction` は暗黙にそこへ登録する。利用者はインスタンスを生成・保持しない | ✅ |
| F5 | マウント可能インスタンス提供 | アクションアプリ `*actions-app*` を `ningle:app` 派生として提供し、本体への mount 統合を可能にする（mount 配線自体は利用者責務・スコープ外） | ✅ |
| F6 | レスポンス整形 | 戻り値の HTTP レスポンス化・content-type 付与・シリアライズ | 対象外(ningle) |
| F7 | htmx レスポンスヘルパ | `HX-Trigger` / `HX-Redirect` / `HX-Retarget` 等の補助（別パッケージ候補） | 将来 |
| F8 | 引数デシリアライズ／型強制 | リクエストパラメータを引数仕様に従ってパース・型変換する | 将来 |
| F9 | 入力バリデーション | 引数仕様に基づく検証と、検証失敗時の統一エラー応答 | 将来 |
| F10 | アクション一覧／レジストリ参照 API | 定義済みアクションの一覧・参照 | 将来 |
| F11 | クライアント呼び出し補助 | フォーム/JS からの呼び出しを助けるヘルパ | 将来 |

※ MVP（初回バージョン）は **F1〜F5** を対象とする。**純粋なアクション機構（定義・登録・ディスパッチ・エンドポイント関数・マウント可能インスタンス）に集中する。**
※ **レスポンス整形（F6）は対象外**。アクション本体の戻り値は ningle の `process-response` がそのまま処理する（文字列→body、Clack リスト→そのまま）。content-type 設定等は利用者が `*response*` で行う。
※ **型強制・バリデーション（F8/F9）は MVP では行わない**。アクション本体は ningle と同じく `params`（alist）をそのまま受け取り、値の取り出し・型変換・検証は利用者の責務とする。
※ **htmx ヘルパ（F7）も MVP 対象外**。`HX-*` ヘッダは ningle 標準の `*response*` ヘッダ操作で記述できるため、コアには糖衣を持ち込まない（将来は別パッケージ候補）。
※ **mount 配線は本ライブラリのスコープ外**であり、利用者が `lack:builder` の `:mount` で行う（ドキュメントで手順を案内）。

## 4. 成功の定義

### 定性的成功基準
- ningle 利用者が、URL の設計・ルート登録の配線を手書きせずに、1つのアクション定義だけでサーバー処理を公開できる。
- アクションを「名前で呼ぶ関数」として参照し、具体的な URL を意識せずに htmx 属性へ埋め込める。
- htmx の `hx-post` から返ってきた HTML フラグメントで UI を更新する、という基本パターンが数行で書ける。
- 既存の ningle アプリへ mount で後付け導入でき、通常のルート定義と URL 名前空間を分けて共存できる。

### 定量的成功基準
- 典型的な「htmx リクエスト受付 → HTML 断片返却」処理の URL 配線・登録の記述量が、素の ningle 実装比で削減される。
- コア機構（F1〜F5）に対するテストカバレッジを確保し、`rove` のテストがグリーンである。

## 5. ビジネス要件
- **ライセンス**: MIT（既存 `.asd` に準拠）。
- **配布**: Quicklisp / Ultralisp で配布可能な単一システム `ningle-actions` として提供。
- **依存方針**: 依存は最小限（ningle と、必要に応じた軽量ユーティリティ）に留め、導入障壁を下げる。
- **後方互換**: ningle の公開 API（`route`、`*request*` 等）のみに依存し、内部実装へ踏み込まない。

## 6. ユーザーストーリー

- **US1**: ningle 開発者として、サーバー関数を「アクション」として定義したい。そうすればエンドポイントの手書き配線をせずに済む。
- **US1a**: 開発者として、アクションを参照すると **エンドポイント URL を返す関数**が得られてほしい。そうすれば具体的な URL を設計・記憶せずにビューへ埋め込める。
- **US1b**: 開発者として、アクションの登録先アプリをいちいち指定したくない。そうすればライブラリのグローバルアプリに自動登録され、記述が簡潔になる。
- **US1c**: htmx 利用者として、アクションごとに HTTP メソッド（GET/POST/PUT/PATCH/DELETE）を選びたい。そうすれば `hx-get` / `hx-delete` 等の使い分けに対応できる。
- **US1d**: 開発者として、アクションアプリを `ningle:app` 派生インスタンスとして受け取り、自分で本体へマウントしたい。そうすれば統合方法を自分で制御できる。
- **US2**: 開発者として、アクション本体で ningle と同じく `params`（alist）を直接扱いたい。そうすれば ningle の知識をそのまま活かせ、独自の引数規約を学ばずに済む。
- **US4**: htmx を使う開発者として、アクションから HTML フラグメントを返して画面の一部を更新したい。そうすれば JSON を介さずハイパーメディアで UI を更新できる。
- **US5**: 開発者として、アクションから `HX-Trigger` 等の htmx 用ヘッダを付けたい。MVP では ningle 標準の `*response*` ヘッダ操作で設定する（専用糖衣は将来の別パッケージ候補）。
- **US6**: 開発者として、アクション内から現在のリクエスト/セッション（`*request*` / `*session*`）へアクセスしたい。そうすれば認証情報などを参照できる。

## 7. 受け入れ条件

- **AC1 (F1/F2/F3)**: `defaction` でアクションを定義すると、(a) ハンドラがシングルトン `*actions-app*` のレジストリに登録され、(b) エンドポイント URL を返す関数が定義される。
- **AC1a (F2)**: アクションアプリは単一の動的ルート `/:action_id` を公開し、`action_id`（ランダム生成の不透明トークン）でハンドラへディスパッチする。
- **AC1b (F1/US1c)**: アクション定義で HTTP メソッドを指定でき、既定は POST。GET/PUT/PATCH/DELETE も指定できる。メソッド不一致のリクエストは 405 となる。
- **AC1c (F3)**: アクションのエンドポイント関数を呼ぶと、`/actions` を前置した完全な URL 文字列（例 `"/actions/<id>"`）が返る。再定義しても URL は安定する。
- **AC2 (F1)**: アクション本体は `(defaction NAME METHOD (PARAMS) ...)` の `PARAMS` を通じて ningle と同一の `params`（alist）を受け取り、`(cdr (assoc "id" params :test #'string=))` 等で値を取得できる。
- **AC3 (F2)**: 未登録の `action_id` へのリクエストは 404 となる。
- **AC4 (F6/ningle)**: アクションが文字列を返すと、ningle の `process-response` によりそれがレスポンスボディになる（ライブラリは戻り値を加工しない）。content-type 等は利用者が `*response*` で設定する。
- **AC5 (US6)**: アクション本体から `*request*` / `*session*` / `*response*` 等の ningle コンテキストにアクセスでき、`HX-*` ヘッダも `*response*` 経由で設定できる。
- **AC6 (F5)**: アクションアプリは `ningle:app` 派生インスタンスであり、利用者が `lack:builder` の `:mount` で本体へマウントできる（mount 配線はライブラリのスコープ外）。

## 8. 機能要件（詳細）

### FR1: アクション定義（defaction）
- 構文: `(defaction NAME METHOD (PARAMS) &body BODY)`。
- アクションは「名前・HTTP メソッド（既定 POST、GET/PUT/PATCH/DELETE も可）・本体」から構成される。URL は利用者が指定せず、ランダムな `action_id` から自動導出する。
- 本体は ningle と同じく `PARAMS`（リクエストパラメータの alist）を直接受け取る。引数仕様・型強制は持たない（MVP）。
- 本体内では `ningle:*request*` `*response*` `*session*` `*context*` も参照できる。
- `defaction` は **エンドポイント URL を返す関数**（`NAME` と同名）を同時に定義する。この関数はキーワード引数を受け取り、そのキー・値を URL エンコードしてクエリパラメータとして付加する（例: `(NAME :category "foo")` → `/actions/<id>?category=foo`）。引数なしの場合は従来どおり `/actions/<id>` を返す。

### FR2: 登録とディスパッチ
- アクションはシングルトンのアクションアプリ `*actions-app*`（`ningle:app` 派生）のレジストリへ登録される。
- アクションアプリは **単一の動的ルート `/:action_id`** を公開し、`action_id` でレジストリを引いてハンドラへディスパッチする。HTTP メソッド不一致は 405、未登録 id は 404。
- `action_id` はランダム生成の不透明トークン。同名アクションの再定義時は既存 `action_id` を再利用し、URL を安定させる。
- 本体 Web アプリへの統合（mount 配線）は **利用者の責務**であり、本ライブラリのスコープ外とする。アクションアプリは `ningle:app` 派生のマウント可能インスタンスとして提供し、利用者が `lack:builder` の `:mount "/actions"` で組み込む。

### FR3: エンドポイント関数生成
- `NAME` と同名の関数を定義し、呼ぶと完全な URL 文字列（`/actions/<action_id>`）を返す。
- 利用者はこの関数の戻り値をビュー（htmx 属性）へ埋め込み、具体的 URL を意識しない。

### FR5: マウント可能インスタンス提供
- アクションアプリは `ningle:app` 派生のマウント可能インスタンスとして提供する。
- 本体への mount 配線は利用者責務（`lack:builder` の `:mount "/actions"`）。

### レスポンスの扱い（対象外: ningle に委譲）
- ディスパッチャはハンドラの戻り値を加工せずそのまま返し、整形は ningle の `process-response` に委ねる（文字列→body、Clack リスト→そのまま）。
- content-type 設定・ステータス・`HX-*` ヘッダは利用者が `*response*` 操作で制御する（ライブラリは関与しない）。

### FR8/FR9（将来）: 型強制・バリデーション
- 将来的に、宣言的な引数仕様（型・必須・既定値）と検証失敗時の統一エラー応答を追加できる余地を残す。MVP では行わない。

## 9. 非機能要件

- **NFR1 (シンプルさ)**: 公開 API は小さく、学習コストを低く保つ。マクロは「薄い」ことを優先する。型強制等の追加機構を持たず、ningle の `params` 流儀を踏襲する。
- **NFR2 (非侵襲性)**: ningle の通常利用を一切妨げず、mount による部分導入が可能であること。
- **NFR3 (テスト容易性)**: `rove` で単体テスト可能であること。シングルトン `*actions-app*` はテスト時に内部 `make-actions-app` で生成した隔離インスタンスへ `let` 再束縛できる。
- **NFR4 (セキュリティ)**: 入力検証は利用者責務だが、ドキュメントで注意喚起する。XSS 対策（出力エスケープ）はビュー層の責務。`action_id` は推測困難なランダムトークンとする。
- **NFR5 (パフォーマンス)**: ルート登録・本体クロージャ生成はマクロ展開時／登録時に済ませ、リクエスト毎のオーバーヘッドを最小化する。
- **NFR6 (移植性)**: 特定の Lisp 処理系固有機能に依存しない（SBCL を主対象とするが移植性を意識）。
- **NFR7 (ドキュメント)**: README とサンプルで、htmx と組み合わせた最小アプリを示す。
