# 要求内容: エンドポイント関数のクエリパラメータ対応

## 背景・目的

現状、`defaction` が生成する同名のエンドポイント関数は引数を取らず、
`/actions/<id>` 形式の URL 文字列のみを返す。

```lisp
(defaction list-items :get (params) ...)
(list-items)  ;=> "/actions/xxxxxxxx"
```

GET 系アクションなどでは、呼び出し側でフィルタ条件やページ番号といった
情報をクエリ文字列としてサーバーに渡したい場面がある。現状はエンドポイント
関数が返す URL に対し、利用者が手作業でクエリ文字列を連結する必要があり、
「URL を意識させない」というプロダクトの価値（URL-free）を損なう。

本作業では、エンドポイント関数がキーワード引数を受け取り、そのキーと値を
**そのまま URL のクエリパラメータとして付加した URL を返す**ようにする。

## ユーザーストーリー

- 開発者として、エンドポイント関数にキーワード引数を渡すだけで、適切に
  エンコードされたクエリ文字列付きの URL を得たい。手作業で文字列連結や
  URL エンコードをしたくない。

## 受け入れ条件

- **AC1**: キーワード引数なしで呼ぶと、従来どおり `/actions/<id>` を返す。
  （後方互換）

  ```lisp
  (list-items)  ;=> "/actions/xxxxxxxx"
  ```

- **AC2**: キーワード引数を渡すと、キー（キーワード名を小文字化した文字列）と
  値が `?key=value` 形式で URL に付加される。

  ```lisp
  (list-items :category "foo")          ;=> "/actions/xxxxxxxx?category=foo"
  (list-items :category "foo" :page 2)  ;=> "/actions/xxxxxxxx?category=foo&page=2"
  ```

- **AC3**: キー・値は URL エンコードされ、安全なクエリ文字列になる
  （スペースや記号、マルチバイト文字を含む値でも壊れない）。

- **AC4**: 文字列以外の値（数値など）も受け付け、文字列表現に変換して付加する。

- **AC5**: 既存の挙動・テスト（再定義での URL 安定性、ハンドラ登録、
  ディスパッチ）に影響を与えない。

## 制約事項

- ライブラリの基本方針「薄さ最優先」（NFR1）を維持する。URL エンコードは
  自前実装せず、既存の依存（ningle/lack 経由で利用可能な `quri`）に委譲する。
- ningle / lack の公開 API のみを使用する方針を維持する。
- サーバー側の `params` 受け取り方（ningle の alist そのまま）は変更しない。
  本変更は「URL を組み立てる側（エンドポイント関数）」のみに閉じる。
- クエリのキー順は呼び出し時の引数順を保持する。
